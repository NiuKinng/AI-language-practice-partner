# 协作约定

## 回复语言

- 默认使用中文回复用户。
- 代码、命令、错误信息和第三方 API 名称按原文保留。

## PR 工作流

- 新功能和修改应通过 PR 合入，不直接在主分支堆叠多个无关改动。
- 每个 PR 只做一件事：只实现或修改单一功能。
- 鼓励尽可能小、粒度尽可能细的 PR；大功能应拆分为多个独立 PR 分步提交。
- PR 合并后，`main` 分支必须保持可运行状态，评委在任意时间查看都应能复现演示效果。
- 合并前至少运行与改动相关的验证命令；涉及页面流程时优先运行 `npm run build`、`npm test`，必要时运行 `npm run e2e`。

## PR 内容要求

- 标题：一句话说明本 PR 新增或修改了什么。
- 功能描述：说明该功能的作用与使用方式。
- 实现思路：简要说明技术选型或核心实现逻辑。
- 测试方式：说明如何验证该功能正常运行。
